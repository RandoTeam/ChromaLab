## Misleading-File Risks

| Risk | Why it matters | Mitigation |
|---|---|---|
| `PHASE9*` docs before 9J look like current truth. | They record intermediate attempts and blocked states. | Use Phase 9J and current validation summary for product truth. |
| DR/Research docs can look like implementation commitments. | They often describe candidates/prototypes. | Treat as method references unless listed in the source-of-truth index. |
| Rust assets under validation look generated but are source-controlled. | They are fixture/parity inputs, not disposable build output. | Keep them; do not clean with ignored artifacts. |
| `artifacts/` contains the most visual evidence but is ignored. | Reinstall or cleanup can remove evidence referenced by docs. | Archive key Phase 9I/9J folders externally before wiping the machine. |
| Grant/public docs can overstate product readiness if read alone. | They are messaging/application materials. | Keep validation truth docs linked near public claims. |

## Cleanup Policy Before Replacement Work

Do:

- keep active source-of-truth docs linked in `docs/README.md`;
- mark old phase docs as historical in new planning docs instead of deleting;
- keep validation assets and Rust parity corpus in source control;
- archive ignored evidence folders externally before reinstall if they are needed;
- delete build/generated directories when disk space is needed.

Do not:

- delete `composeApp/src/androidMain/assets/validation/`;
- delete `rust/` source files;
- delete `tools/benchmark/` or `tools/rust/` while Rust replacement work is active;
- commit model binaries or signing keys;
- use old phase docs as implementation authority without current-code verification.

## R0 Conclusion

The repository is not blocked by a lack of documentation. It is blocked by too
many historical documents and partially overlapping runtime paths. Replacement
work must start from the owner board, not from another standalone prototype.
